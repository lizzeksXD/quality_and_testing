1. Архитектурный: Используем BookRepo как посредника. Контроллер не работает напрямую с базой данных, а общается только с репозиторием. 

2. Технический: db.session.delete(book) без commit(). изменения будут в транзакции, но не сохранятся в БД. При закрытии сессии изменения откатятся.

3. HTTP: Post нарушает семантику HTTP:
•	По стандарту, удаление должно выполняться методом DELETE.
•	POST предназначен для создания или изменения ресурсов, а не их уничтожения.
•	Такой подход усложняет построение RESTful API в будущем.
Правильно использовать DELETE метод, который семантически соответствует удалению ресурса.

4. Безопасность: Без CSRF возможна атака, когда злоумышленник заставляет пользователя удалить книгу без его ведома. Некому наши книжки воровать и использовать)

5. Проектирование:
Без JavaScript: можно добавить страницу "Вы уверены?" с кнопкой "Да и Нет"
С JavaScript: всплывающее окно "Удалить книгу? ОК или Отмена"

6. SQLAlchemy: get_or_404() проверяет, найдена ли книга. Если нет, то сразу показывает ошибку 404. С обычным get() нужно каждый раз проверять результат и вручную обрабатывать случай, когда книга не найдена.
